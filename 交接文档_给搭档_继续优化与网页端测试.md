## 1. 怎么把它跑起来（两种模式，先搞清区别）

环境关键事实（这块踩过坑，务必看）：详见 `memory` 与 `CONTEXT_HANDOFF.md §2`。

### 模式 A：纯前端 mock（最快，改 UI 用这个）
不需要后端，所有数据走 MSW 假接口。
```bash
cd apps/web
VITE_ENABLE_MSW=true npm run dev
# 打开 http://localhost:5180/
```
适合：调 UI、调交互、跑 QA 脚本里写的那些 demo 场景（`docs/QA_TEST_RECORD.md` 里的 URL 都是 5180 mock 端口）。

### 模式 B：真实后端全栈（测真实链路用这个）
Docker 起 web + api + worker + mysql + redis。**当前这套已经在跑**（我刚确认 5 个容器都 Up）。
```bash
docker compose up -d            # 若没起
# 打开 http://localhost:5173/   ← 注意是 5173，不是 5180
```
- **端口别记错**：`5180` = 本地 mock dev；`5173` = Docker 真实后端。
- **后端没有热更新**：api/worker 跑的是打包进镜像的 `/app` 代码，改后端代码必须 `docker compose up -d --build api worker` 才生效。但你基本不碰后端，知道就行。
- web 容器是 vite dev，挂载源码、HMR 生效，改前端会自动刷新。

### 改后端代码的工作流（你要动 `apps/api/` 必看 ⚠️）
后端有几个非直觉的坑，踩过，记牢：
- **api/worker 没有热更新**：它们跑的是打包进镜像 `/app` 的代码，**改了源码不会自动生效**。要让运行中的后端生效：
  ```bash
  docker compose up -d --build api worker
  ```
- **但 pytest / seed / 同步脚本走 `/workspace` 挂载，读当前源码**，改测试或 seed 不用重建镜像：
  ```bash
  docker compose exec -w /workspace/apps/api api pytest -m "not integration"   # 165 passed
  docker compose exec -w /workspace/apps/api api bash scripts/e2e_test.sh        # E2E
  ```
- **改了数据库模型必须走迁移，别手改库**：
  ```bash
  docker compose exec -w /workspace/apps/api api alembic upgrade head
  ```
- 技术栈：Python 3.11 + FastAPI + SQLAlchemy + Celery + Redis + MySQL。本机系统 Python 3.9 太旧跑不了后端，**测试统一在容器里跑**（上面的命令）。
- 路由风格：FastAPI `APIRouter`，`/api/v1` 前缀在挂载处统一加，domain 逻辑在 `app/services/*_domain.py`，Pydantic schema 在 `app/schemas/*`，审计走 `audit_event_domain.emit_audit_event(...)`。
- 真实 LLM 链路已配好（DOUBAO key 在根 `.env`，模型 `ep-20260514105718-jthdm`），可真实跑 AI 预审。

更细的后端/环境上下文见 `CONTEXT_HANDOFF.md §2`、`CLAUDE.md`、`AGENTS.md`。

## 2. 项目结构（你需要知道的部分）

```
apps/web/                    ← 你的主战场（React + Vite）
  src/features/
    owner/                   出题、Schema 模板搭建、AI 预审配置、质量中心、导出
    labeler/                 任务广场、标注工作台、我的提交
    reviewer/                审核详情、AI Assist 面板、diff
  src/mocks/                 MSW：handlers.ts + mock-db.ts + data/*.mock.ts
```

## 3. 现在做到哪了（已完成 vs 待优化）

### ✅ 已经真实跑通的（别破坏它们）
- **全链路闭环**：Owner 建任务→搭 Schema 模板→导入数据集→配 AI 预审→发布；Labeler 领取→作答（含 Formily 智能联动、草稿自动保存、题目级 AI 辅助）→提交→「我的提交」看状态/打回；Reviewer 审核 diff + AI Assist 一键采纳；Owner 质量中心 + 导出质量护照。
- **Schema Runtime（核心，最容易被 UI 改动误伤）**：
  - `qualityScore = 1/2` → `factCheckNote` 显示并必填；`= 3/4/5` → 隐藏并清空。
  - 提交校验按 runtime 状态判断：`visible+required+empty` 才阻断，`hidden/disabled required` 不阻断。
  - AI preflight：suggestedPatch 不直接写答案，patch 引入新必填缺失时禁止一键采纳。
- **Reviewer AI Assist 闭环**（最近一轮）：一键采纳 / 编辑后采纳 / 忽略，写真实审计事件，终态不可重复采纳。
- **Owner 质量中心**：总览 10 项真实统计 + 6 大看板，全部实时计数（不写死），只展示人话摘要、不泄漏 raw event code / JSON。

### ⚠️ 已知待优化 / 待排查（你的活儿）
来自前一份交接文档 `LabelHub_前端_UI_功能交接文档_更新优化版.md §1.2`，逐条核实是否仍存在：

| 模块 | 问题描述 | 建议动作 |
|---|---|---|
| 任务发布页 | 历史上出现过「创建/发布失败」 | 真实后端走一遍发布，看 payload 是否符合 contracts、错误是否人话、失败有无页面反馈 |
| AI 预审页面 | 紫/橙跳色过重，风格不统一 | 统一 Corporate Trust 风格；清晰表达异步队列/维度评分/失败重试/人工兜底 |
| Labeler 任务广场 | 三类任务（广场先到先得 / 配额抢单 / 指派给自己）混成一个列表 | 拆分展示，别混 |
| 标注台 | 题目导航、草稿保存反馈、打回提示、题目级 AI、提交校验仍可打磨 | 逐项体验，补反馈 |
| 人工审核流转 | 复审/终审视图、第 1/2 轮 diff、批量操作、完整审计时间线待完善 | Reviewer 只看与自己审核相关的信息 |
| 死按钮 / 假反馈 | 历史上有写死的假时间戳、点了没反应的按钮 | 按 §5 死按钮清单逐页扫 |

## 4. 网页端逐步测试指南 ⭐（重点）

> **怎么用**：照下面顺序一个角色一个角色地点，每点一步对照「预期」，不符就记到第 6 节缺陷表。
> mock 模式 demo 场景齐全（推荐先用 5180 把 UI 行为测全），再用 5173 真实后端验证关键链路。
> **现成的细颗粒清单在 `docs/QA_TEST_RECORD.md`** —— 那份是带 URL + 逐项勾选 + 截图命名规范的正式 QA 表，**你测的时候直接填那张表**，本节只讲整体顺序和该盯什么。

### 测试前置（每次开测先做）
1. 确认分支/状态：`git rev-parse --abbrev-ref HEAD`（应为 `integration/joint-test`）、`git status`。
2. 跑一遍自动化（应全绿）：
   ```bash
   cd apps/web && npm run typecheck && npm run build
   ```
   后端/其它包测试命令见 `docs/QA_TEST_RECORD.md §3`。
3. 起服务（模式 A 或 B），打开页面，**先看浏览器 Console 有没有红色 error**——整个测试过程 Console 都要盯着。

### 测试顺序（按角色走，对应 QA 表的章节）

**① Owner 出题 / Schema 治理**（QA 表 §4）
- 建任务 → 搭 Schema 模板（左物料 / 中画布 / 右属性·校验·联动）→ 导入数据集 → 配 AI 预审 → 发布。
- 重点看：Breaking Change 是否阻断发布、安全发布是否成功、Audit Timeline 事件是否正确、发布失败有没有人话提示。
- demo 场景 URL（mock）：`/owner/tasks/task_demo_schema_breaking_change/designer` 等，见 QA 表 §4.2~4.5。

**② Labeler 标注**（QA 表 §5、§6）
- 工作台 `/labeler/workspace/asn_1001`。
- 重点看：
  - `qualityScore` 选 1 → `factCheckNote` 出现且必填；选 3/4/5 → 隐藏并清空（**这是核心联动，必测**）。
  - 必填没填时提交按钮不误提交；填完整能提交成功且有页面反馈。
  - 草稿自动保存：编辑后顶栏徽章显示真实「保存中 / 已自动保存 HH:MM:SS」（**不能是写死的假时间**）。
  - AI 辅助按钮可点，BLOCKED 场景「确认应用建议」物理禁用、SAFE 场景可应用。
  - **普通 UI 不能出现工程词**：`legacy` / `formily-v2` / `mock` / `raw patch` / `preflight blocked` / `answers JSON` 等（完整禁词表见前端交接文档 §5.4）。
- 「我的提交」页 `/labeler/submissions`：按状态归类、打回项能跳回工作台改。

**③ Reviewer 审核**（QA 表 §7）
- `/reviewer/items/sub_1001`（无修改审核，不产生 diff）、`sub_1003`（改答案 → 产生 `REVIEW_DIFF_GENERATED`）、`sub_1004`（AI Review 反馈）。
- 重点看：AI Assist 面板「一键采纳 / 编辑后采纳 / 忽略」能用、采纳后状态变「已采纳」且不可重复、审计事件实时出现、diff payload 只含字段名不含完整答案。

**④ Owner 质量中心 + 导出**（QA 表 §8）
- `/owner/quality`：总览统计是否真实、6 大看板是否齐全、有没有泄漏 raw JSON。
- `/owner/tasks/.../export`：导出生成质量护照，`passportBatchHash` 应为真实 SHA-256（不是 `sha256:mock-xxx`）。

### 每页都要顺手检查的通用项（死按钮 / 反馈清单）
- [ ] 有没有按钮点了没反应；未实现的功能应 disabled + tooltip，不留假死按钮。
- [ ] 提交成功/失败有页面反馈，不只在 Console。
- [ ] 没有写死的假时间戳。
- [ ] hash / 审计时间线 / 长文本不溢出。
- [ ] 普通 UI 无工程词（见上）。
- [ ] Console 全程无 `submit 500` / `audit-events 500` / 红色 error。

## 5. 继续优化的优先级（建议顺序）

> 原则：**先保证真实链路能跑，再做 UI polish；先把人能看懂的 AI 建议做好，再做更漂亮的界面。**

1. **P0 不破坏主链路**：每次改完跑 `npm run typecheck && npm run build`，并手测 Labeler 联动 + 提交 + AI preflight（§4②）。
2. **P1 任务发布失败排查**：这是历史遗留最该先确认的功能性问题。
3. **P1 AI 预审页面风格统一**：去掉过重紫/橙，统一 Corporate Trust。
4. **P2 Labeler 任务广场拆分**：三类任务分开展示。
5. **P2 标注台交互打磨**：题目导航 / 草稿反馈 / 打回提示 / 提交校验。
6. **P2 人工审核流转补全**：复审/终审、批量、审计时间线。
7. **P3 死按钮 + 视觉统一**：字体/字号/溢出/disabled 视觉。
8. **最后**：补 QA 截图 + 测试记录。

各模块的详细 UI 规格（物料清单、画布要求、属性/校验/联动配置、AI Panel 推荐文案结构）已经写在 `LabelHub_前端_UI_功能交接文档_更新优化版.md §6~§10`，做之前对着那份做，别重新设计。

## 7. 工作方式与纪律（重要）

> 自 2026-06-09 起，项目**全权交给你**，原负责人（魏）停止迭代。你是唯一开发者。

- **可以直接在 `integration/joint-test` 主线上迭代，不用开分支。** 既然只有你一个人在改，没有并发踩踏问题，开分支反而徒增流程。
- **但守住一条安全网——保留稳定基线 tag**：交接时已打 `handoff-stable-0609`（全绿、能 demo 的干净状态）。万一哪天把主线改乱了、答辩前慌了，`git checkout handoff-stable-0609` 就能回到今天这个版本。每到一个新的「稳定可 demo」节点，建议自己再打一个 tag（`git tag -a demo-stable-MMDD -m "..."`）。
- **push 前两条铁律**：① 先 `git pull --rebase`（保持与远程一致）；② **绝不 push 跑不起来的 commit**——push 前至少 `cd apps/web && npm run build` 过一遍。坏改动进主线就没人帮你兜底了。
- **改契约/接口/字段名**：必须 `packages/contracts → apps/api/app/schemas/*.py → apps/web` 三处同步改，别只改一半，改完前后端测试都要跑过。
- 提交前自检：
  - 前端：`cd apps/web && npm run typecheck && npm run build`；动了 schema-renderer/designer 再跑各自的 `npm run typecheck`。
  - 后端：容器内 `pytest -m "not integration"` + `bash scripts/e2e_test.sh`；改了 DB 模型要写 alembic 迁移。
- 频繁、小步提交，commit 信息写清「改了什么 + 验证结果」，方便日后自己回溯/回滚。
